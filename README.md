## Project Overview
DataParq Automation is a robust, data-driven UI test automation framework built using Python, Selenium WebDriver, and Pytest. It is specifically designed to automate end-to-end data workflows, validation pipelines, and governance tracking on the DataParq platform. The framework leverages the Page Object Model (POM) design pattern to maintain modular, scalable, and highly readable test scripts.

## Features
- **Page Object Model (POM):** Separates UI interaction logic from test logic for improved maintainability.
- **Fail-Fast End-to-End Pipeline:** Contextually passes test states (like generated Domain Names) between sequential test cases via `run_info.json`.
- **Dynamic File Uploads:** Validates data uploads with configurable inputs from the CLI.
- **Automated Reporting:** Generates rich `.docx` Execution Reports on-the-fly documenting every test step.
- **Visual Evidence:** Automatically captures screenshots at every workflow step and embeds them in the reports.
- **Custom Kendo UI Interactions:** Handles complex third-party grid controls, multi-select dropdowns, and dynamic filter overlays.
- **Background Task Monitoring:** Long-polling strategies to dynamically wait up to 10 minutes for asynchronous backend tasks to resolve.

## Tech Stack
- **Language:** Python 3.8+

## Prerequisites
Before you begin, ensure you have the following installed:
- Python 3.8 or higher
- Google Chrome (latest version)
- Git

## Installation Steps
1. **Clone the repository:**
   ```bash
   git clone <repository_url>
   cd dataparq_automation
   ```

2. **Create a virtual environment:**
   ```bash
   python -m venv venv
   ```

3. **Activate the virtual environment:**
   - Windows: `venv\Scripts\activate`
   - Mac/Linux: `source venv/bin/activate`

4. **Install dependencies:**
   ```bash
   pip install -r requirements.txt
   ```

## Environment Setup
Create a `.env` file in the root directory based on the following template to securely store your credentials:
```env
BASE_URL=https://<your-dataparq-instance-url>
USERNAME=your_username
PASSWORD=your_password
```

## Configuration Details
The framework is configured via `pytest.ini` which sets default CLI arguments and logging settings. 
Browser sessions are globally managed in `conftest.py`. The `driver` fixture is set to `session` scope to allow sequential Test Cases (e.g. TC1 -> TC2 -> TC3) to reuse the same authenticated browser session.

## Running the Framework

### Running Individual Test Cases
To run a specific test case (for example, Test Case 3):
```bash
pytest tests/test_tc03_Verify_upload_file_in_Monitor_Module.py -v
```

### Running the Regression Suite (End-to-End Pipeline)
The test suite is designed to pass context (like the generated `workflow_name`) across files. Execute them in sequential order:
```bash
pytest tests/test_tc01_file_upload.py tests/test_tc02_data_governance.py tests/test_tc03_Verify_upload_file_in_Monitor_Module.py --upload-file "test_data/international_cricket_match_analytics.csv" -v -x
```
*(The `-x` flag enables fail-fast mode, immediately halting the suite if a dependency fails).*

## Test Execution Flow
1. **conftest.py** initializes the Chrome WebDriver and clears stale `reports/` and `run_info.json` files.
2. **TC1** logs in, uploads the dataset, captures the generated Domain Name, and writes it to `run_info.json`.
3. **TC2** inherits the browser session, navigates to Data Governance, and searches using the generated Domain Name.
4. **TC3** inherits the browser session, filters the Data Lakehouse by the Domain Name, clicks the Task icon, and polls the backend Task Engine for up to 10 minutes until status reaches `Completed`.
5. **conftest.py** tears down the session and finalizes Word documents in the `reports/` folder.

## Reporting and Logs
The framework automatically logs all console outputs and saves them locally.
- **Screenshots:** Saved dynamically in `reports/screenshots/`
- **Execution Reports:** `.docx` files detailing each test step and its visual evidence are compiled in `reports/execution_reports/`
- **System Logs:** Terminal output is printed using standard Python `logging`.

## CI/CD Integration
To integrate with Jenkins, GitHub Actions, or GitLab CI:
1. Ensure the CI environment supports Headless Chrome. Update `conftest.py` `ChromeOptions` to include `--headless=new` if running on a Linux build agent without a display.
2. Execute the Pytest command as a build step.
3. Archive the `reports/execution_reports/` folder as a build artifact.

## Troubleshooting Guide
- **`TimeoutException` during execution:** Ensure that the target UI element is visible. DataParq grids may occasionally take longer to load; you can increase `WebDriverWait` limits in the respective page objects.
- **`WinError 32` when cleaning up reports:** A previous test run may have crashed, leaving a file handle open. Close any open `.docx` reports or orphaned `chromedriver.exe` processes in Task Manager.
- **Missing `run_info.json`:** Ensure you run TC1 before running TC2/TC3, as TC1 is responsible for generating this state file.
